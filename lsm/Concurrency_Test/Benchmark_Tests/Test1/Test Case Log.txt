Multiple Write then Read then Write operations 

Expected: The first transaction has a write lock on the file. All other transactions will wait in order until this write lock is removed. So T0 Run first then T1 and so on and so forth. 
Actual: T0's first write put a lock on the record causing all reads and writes to be put on hold. Since these locks block any other operations, we put these incoming operations in the transaction delay list. Once the transaction was allowed to run again after T0 released it's locks, the later transactions were able to acquire locks to run all it's operations and commited. In the end, all operations ran in serial execution and no dirty reads.
